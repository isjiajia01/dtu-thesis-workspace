# 22.03controller_line vs 22.04fresh_solver：关系与口径说明

## 1. 目的
本文档用于说明 `22.03controller_line` 与 `22.04fresh_solver` 两套系统之间的关系、研究定位差异、统计口径映射，以及论文中如何同时安全引用两者。

目标不是把两套系统“强行写成同一套方法”，而是：
1. 明确二者各自解决的问题；
2. 提供可跨系统对照的顶层 KPI 口径；
3. 明确哪些指标可以比较，哪些指标不能直接横向比较；
4. 为论文写作提供安全表述边界。

---

## 2. 两套系统的关系：不是简单升级关系，而是两条研究路径

## 2.1 22.03controller_line 的定位
`22.03controller_line` 更适合被理解为：
- 一个 **controller / policy-centered** 的 rolling-horizon 实验系统；
- 重点在于研究在压力场景（如 `EXP01` / `Scenario1`）下，controller 机制如何改进服务率与失败数；
- 日内 routing backend 更像一个 fixed-budget execution module，而不是论文的主要创新载体。

它的主要优点是：
- 论文叙事成熟；
- baseline / pressure / method line 的结构清晰；
- `claim guardrails` 完整；
- `main / supplementary / historical` 的边界已经被明确约束。

## 2.2 22.04fresh_solver 的定位
`22.04fresh_solver` 更适合被理解为：
- 一个 **solver / architecture-centered** 的新求解系统；
- 重点在于围绕 thesis 当前问题定义，重新搭建 `controller + daily routing backend + depot-aware repair`；
- 把 gate / picking / staging / depot overload / route repair 等更直接地纳入求解链本身。

它的主要优点是：
- 更贴近 thesis 当前确认的问题结构；
- 更接近“真正的求解器系统”，而不只是 policy simulation；
- 更能承载仓配耦合、depot-resource realism、route repair 等新研究点；
- Julia-first，更符合当前主实现方向。

## 2.3 二者关系的正确理解
不建议将二者理解为：
- `22.04` 是 `22.03` 的简单升级版；
- 或 `22.03` 完全被 `22.04` 否定。

更准确的理解是：
- `22.03`：成熟的 controller-centered 研究与论文叙事参考系；
- `22.04`：当前 thesis 主求解架构的重建与新实验主线。

| 维度 | 22.03controller_line | 22.04fresh_solver |
| --- | --- | --- |
| 主研究对象 | controller / policy line | integrated solver architecture |
| 系统结构重心 | rolling simulation + controller + daily backend | controller + routing constructive + depot-aware repair |

| 当前适合承担的角色 | reference line / writing guardrail | thesis 主算法主线 |

## 5. 哪些指标可以比较，哪些不能直接比较

## 5.1 可以跨系统比较的顶层 KPI
以下指标可以作为 `22.03` 与 `22.04` 的 paper-facing 可比指标：
- `eligible_count`
- `delivered_within_window_count`
- `deadline_failure_count`
- `service_rate_within_window`

原因：
- 都是 unique-order level 的 horizon summary；
- 都可被解释为“窗口内可服务订单中，最终按时完成了多少”；
- 都不依赖系统内部某个具体 repair / buffer / controller trace 的细节。

## 5.2 不建议直接横向比较的过程指标
以下指标不应直接在两套系统间做一一对照：

### carryover
- `22.03` 中 carryover 更接近 `planned but not delivered` 的 pool；
- `22.04` 中 carryover 更接近 `deferred / unassigned but still feasible next day` 的 pool。

### deferred / rejected / unserved breakdown
- `22.03` 会进一步区分 `policy_rejected_or_unserved`、`vrp_unserved_on_deadline` 等；
- `22.04` 当前有 `controller deferred`、`routing unassigned`、`repair rollback` 等不同来源。

### cost / penalized_cost
- `22.03` 的 `penalized_cost` 是 paper-facing 核心指标；
- `22.04` 当前更强调 `depot_penalty`、`overload_bucket_count` 等执行风险信号；
- 两边的 cost semantics 目前不一致，不能直接做数值排名。

### depot_penalty / overload metrics
- 这是 `22.04` 的新增研究维度；
- `22.03` 没有同语义的对等指标。

## 5.3 一个重要原则
**口径可对齐 ≠ 结果可直接排名。**

即使两个系统都输出了：
- `eligible_count`
- `delivered_within_window_count`
- `deadline_failure_count`
- `service_rate_within_window`

也不能自动说明它们是在相同实验世界下产生的结果。因为：
- `22.03` 更偏 controller/policy simulation；
- `22.04` 更偏 integrated solver；
- 二者在 routing backend、depot realism、repair logic、场景定义上都不同。

因此，统一口径的意义在于：
- 统一表达；
- 统一表格；
- 统一 safe claims；

而不是把两套系统混成一个排名榜。

## 6. 论文里如何同时引用两套系统

## 6.1 推荐角色分工
### 22.03controller_line 在论文中的角色
建议将 `22.03` 写成：
- 前一阶段成熟的 controller-centered reference line；
- 提供安全的 paper-facing KPI 语言；
- 提供 baseline / pressure / controller improvement 的成熟叙事模板；
- 作为 safe claim 与 retained endpoint 的组织参考。

### 22.04fresh_solver 在论文中的角色
建议将 `22.04` 写成：
- thesis 当前主研究系统；
- 用于呈现 `controller + daily routing backend + depot-aware repair` 的新架构；
- 用于分析 optimistic baseline vs strict baseline；
- 用于分析 Herlev / East / West 的 regime difference。

## 6.2 推荐写法
### 可以写成
- `22.03` established a controller-centered rolling-horizon reference line with paper-facing service/failure metrics.
- `22.04` rebuilds the thesis main algorithmic architecture around an integrated controller-routing-repair solver.
- For cross-system readability, top-level service metrics are aligned through a common paper-facing schema (`eligible_count`, `delivered_within_window_count`, `deadline_failure_count`, `service_rate_within_window`).

### 不建议写成
- `22.04` directly outperforms `22.03` on the same experimental basis.
- `22.03` and `22.04` are two versions of the same solver and can be ranked in one flat table.
- identical service-rate numbers imply identical operational meaning.

---

## 7. 推荐的论文表格组织方式

## 7.1 可以放在同一节，但分两张表
### 表 A：22.03controller_line paper-facing retained results
- 只展示 `22.03` 的 retained controller line / OOD line；
- 用 `eligible_count`, `delivered_within_window_count`, `deadline_failure_count`, `service_rate_within_window`, `penalized_cost` 组织。

### 表 B：22.04fresh_solver integrated solver results
- 展示 `22.04` 的 optimistic baseline / strict baseline；
- 用相同的顶层 KPI 展示：`eligible_count`, `delivered_within_window_count`, `deadline_failure_count`, `service_rate_within_window`；
- 同时增加 `22.04` 特有的过程指标：`total_deferred_events`, `depot_penalty`, `overload_bucket_count`。

## 7.2 不建议做的表格
不建议把：
- `22.03 main candidate`
- `22.04 optimistic baseline`
- `22.04 strict baseline`
- `22.03 OOD east/west`
- `22.04 East/West`

全部塞进一张“最终 best model ranking”表。

这是因为：
- 场景不同；
- 系统不同；
- realism 口径不同；
- ranking 语义会混乱。

## 8. 当前可执行的安全结论

## 8.1 可以安全说
1. `22.03` 与 `22.04` 分别代表两条不同研究路径：controller-centered 与 integrated-solver-centered。
2. `22.04` 已经补齐与 `22.03` 对齐的 paper-facing summary fields，因此两套系统现在可以在顶层 service/failure 指标上进行统一表达。
3. `22.04` 的 depot-aware metrics（如 depot penalty / overload buckets）是其新增研究价值，不应因追求口径统一而被删除。
4. 两套系统可以在同一论文中并存，但应承担不同叙事角色。

## 8.2 不应直接说
1. `22.04` 已经在同一实验基准下严格超过 `22.03`。
2. 两套系统的所有指标都可以一一对齐。
3. `22.04` 当前 optimistic baseline 可以直接替代 `22.03` 的所有 paper-facing claims。

---

## 9. 当前建议
1. 继续用 `22.03` 作为写作口径、claim guardrails、retained endpoint 组织方式的参考。
2. 继续用 `22.04` 作为 thesis 当前主算法系统推进。
3. 在所有 `22.04` 结果输出中保留两层指标：
   - paper-facing aligned metrics；
   - solver-native process metrics。
4. 论文写作时，明确区分：
   - cross-system comparable KPI；
   - system-specific diagnostics。

---

## 10. 一句话总结
`22.03` 提供的是**成熟的 controller-centered reference language**，`22.04` 提供的是**当前 thesis 主架构的 integrated solver system**。两者应在顶层 KPI 上对齐表达，但不应被误写成同一实验世界下的平行模型排名。
