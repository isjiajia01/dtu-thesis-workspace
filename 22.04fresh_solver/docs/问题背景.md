# 问题背景

本项目研究的是一个**多日滚动规划的末端配送问题**。业务不是一次性拿到全部订单后做静态排程，而是每天在已有已释放订单、未完成订单和当天仓配能力约束下重新制定配送计划。订单既可能是固定送达日，也可能在若干可行日期内灵活安排，因此问题同时包含“哪一天送”与“当天怎么送”两层业务决策，但这里仅描述**业务背景、问题边界、数据规模与约束定义**，不涉及算法架构或求解实现。

## 1. 业务场景
- 规划以天为单位滚动进行，记作 `t = 1..T`。
- 每天决策时，只能看到**已释放但尚未完成**的订单；未来尚未释放的订单在当天不可见。
- 订单可以带有单日或多日服务窗口，因此系统需要在允许日期内安排其服务日。
- 当天执行后，未完成的订单会进入后续日期，形成跨日累积压力。
- 这是一个真实仓配环境下的问题，而不是纯粹的单日标准 VRP；因为除了客户点配送，还要同时满足仓库出车、装卸、备货与车辆资源限制。

## 2. 订单与规划对象
每个订单至少包含以下业务信息：
- `Depot`：订单所属仓库/站点。
- `DateRequested`：订单释放日；在释放日前订单不可见、不可分配。
- `DeliveryDateFrom`：最早可服务日期。
- `DeliveryDateFrom2`：最晚可服务日期。
- `DeliveryEarliestTime` / `DeliveryLatestTime`：客户时间窗，限制当天服务开始时刻。
- `ColliCount`、`Volume`、`Weight`：三维需求量。
- `DeliveryTaskTime`：客户现场服务时长。
- `PickingTaskTime`：仓内拣货工作量。
- `PickupAddress` / `DeliveryAddress` 及对应经纬度：用于节点映射与路网矩阵关联。

据此，每个订单都带有一个**硬性的跨日可服务窗口**与一个**硬性的日内时间窗**。

## 3. 仓库与车队背景
仓库侧数据来自 `Warehouses` 工作表，车辆侧数据来自 `Vehicles Settings` 工作表。问题中会同时考虑：

### 仓库侧业务含义
- `EarliestArrival`：当天最早允许车辆出车的时刻。
- `LatestDeparture`：当天最晚允许车辆离仓出发的时刻；它表示**最后允许出车时间**，不是最晚回仓时间。
- `PickingOpeningHours` / `PickingClosingHours`：仓内拣货可作业时间窗。
- `Gates`：仓库同一时间段内允许的出车门位/平台数。
- `VehicleLoadingTime` / `VehicleUnloadingTime`：车辆装货与回仓卸货耗时。
- `MaxStagingSpace`：暂存区容量上限。
- `PickingCapacityColliPerHour` / `PickingCapacityVolumePerHour`：每小时拣货吞吐能力。
- `Return (Yes/No)`：是否要求车辆回仓；当前确认的数据口径为 `Yes`，因此配送任务需回仓闭环完成。

### 车辆侧业务含义
- `Number of vehicles`：某仓某车型每天可用车辆数。
- `CapacityColliCount`、`CapacityVolume`、`CapacityWeight`：车辆三维容量上限。
- `MaxRouteDuration`：单趟路线最大工作时长。
- `MaxRouteDistance`：单趟路线最大里程。
- 当前业务默认：**每台物理车辆每天最多执行两趟**；若执行第二趟，则必须在第一趟完成且完成卸货与再装货后才能再次出发。

## 5. 路网与时空口径
- 配送时间与距离不是用欧氏距离近似，而是使用**真实路网矩阵**。
- 主基准 Herlev 对应 `data/processed/vrp_matrix_latest/`。
- 其他分仓数据集分别对应各自矩阵目录，例如 east / west / aalborg / aabyhoj / odense。
- 因此，客户访问顺序、车辆行驶时间、到达可行性与总里程都以路网矩阵为准。

## 6. 必须保持一致的业务约束
以下约束是当前文件口径下需要严格保持一致的内容。

### 6.1 订单可服务性约束
- 每个订单只能由其所属 `Depot` 的车队服务，不能跨仓服务。
- 订单在 `DateRequested` 之前不可见、不可服务。
- 订单只能在 `[DeliveryDateFrom, DeliveryDateFrom2]` 对应的可行日期集合内被服务。
- 业务目标是每个订单在其可行窗口内完成且仅完成一次服务。
- 若采用允许软违约的建模口径，则未服务订单必须被显式标记，并进入失败、延期或未服务统计，而不是隐式消失。

### 6.2 客户时间窗与服务时长约束
- 客户服务开始时刻必须满足 `DeliveryEarliestTime <= service_start <= DeliveryLatestTime`。
- `DeliveryTaskTime` 必须计入当天行程持续时间。
- 因此，客户时间窗不是偏好，而是硬约束。

### 6.3 车辆容量与单趟执行约束
- 每条配送路线都必须同时满足三维容量限制：
  - `ColliCount`
  - `Volume`
  - `Weight`
- 每趟路线都必须满足：
  - `MaxRouteDuration`
  - `MaxRouteDistance`
- 当前数据口径下，配送任务要求回仓，因此路线需闭合回到仓库。

### 6.4 车辆数量与多趟约束
- 每天每个仓、每个车型可用车辆数受 `Number of vehicles` 限制。
- 默认每台物理车辆每天最多两趟。
- 第二趟必须晚于第一趟结束，并预留：
  - `VehicleUnloadingTime`
  - `VehicleLoadingTime`
- 因而，多趟执行不是无限追加，而是受同一车辆时间轴约束。

### 6.5 仓库出车与门位约束
- 所有车辆出发都必须满足 `EarliestArrival` 与 `LatestDeparture` 给出的出车窗口。
- `LatestDeparture` 约束的是**离仓出发时刻**，不是回仓完成时刻。
- `Gates` 是硬约束：在离散时间口径下，同一 `15` 分钟 bucket 内的出车数不能超过门位数。
- 这意味着即使车辆和订单本身可行，也可能因为同一时段仓库平台资源不足而不可同时执行。

### 6.6 拣货、暂存与仓内吞吐约束
- `PickingTaskTime` 表示仓内拣货工作量，本质上对应仓库资源负荷，而不是客户点上的日内服务时长。
- 拣货作业只能发生在 `PickingOpeningHours` 到 `PickingClosingHours` 之间。
- 仓内作业能力受以下上限约束：
  - `PickingCapacityColliPerHour`
  - `PickingCapacityVolumePerHour`
  - `MaxStagingSpace`
- 在 thesis 当前建模口径中，这部分约束作为 warehouse-resource 输入，并在启用相应仓库资源约束时生效。
- 业务上允许夜间预拣/预装，但正式出车仍必须满足当天出车窗口与门位限制。

## 7. Herlev 主基准的已确认仓库口径
对于当前 thesis 主 benchmark Herlev，已确认的仓库参数为：
- `EarliestArrival = 06:00`
- `LatestDeparture = 17:00`
- `PickingOpeningHours = 00:00`
- `PickingClosingHours = 23:59`
- `Gates = 10`
- `MaxStagingSpace = 500`
- `PickingCapacityColliPerHour = 200`
- `PickingCapacityVolumePerHour = 100`

这组口径的业务解释是：
- 夜间可以进行拣货与暂存；
- 但车辆正式离仓仍不能早于 `06:00`；
- 最晚离仓时间不能晚于 `17:00`；
- 同时离仓能力受 `10` 个门位限制。

## 8. 问题目标的业务表达
在上述真实业务条件下，规划问题希望同时做到：
- 尽量提升订单按时履约率；
- 尽量减少逾期、失败与跨日积压；
- 在满足仓库与车辆约束的前提下控制总行驶时间/距离；
- 在滚动重规划场景下避免不必要的计划剧烈波动。
