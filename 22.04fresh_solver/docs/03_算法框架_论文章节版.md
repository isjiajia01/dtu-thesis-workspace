# 第3章 算法框架设计

## 3.1 Overall Framework

针对本研究中的多日滚动规划仓配一体化多趟配送问题，本文不采用一个单体化的全局精确模型，而是提出一个**分层滚动启发式框架**。其核心思想是将原问题拆解为三个相互耦合但职责清晰的层次：

1. **Rolling Controller（跨日控制层）**：决定在规划日 `t` 哪些订单应被纳入当天执行，哪些订单应被保护，哪些订单应被延后。
2. **Daily Rich VRP Backend（单日富约束路径后端）**：对当天被接纳的订单集生成多趟、带时间窗和多维容量约束的日内车辆路径。
3. **Depot-aware Repair Layer（仓库资源修复层）**：在保持日内路线主体结构尽量稳定的前提下，进一步修复门位、拣货、暂存与 trip2 高峰等仓库执行层冲突。

整体框架运行于 daily rolling horizon 上。对于每个规划日 `t`，系统读取当前所有**已释放但尚未完成**的订单，生成当日方案，并在执行结束后将未完成订单与状态信息传递给下一规划日 `t+1`。因此，该框架是一种**状态驱动的序列决策系统**，而非一次性静态优化。

### 3.1.1 Why a layered framework is necessary

本问题同时包含以下三个彼此不同的决策尺度：

- **跨日尺度**：某订单应在今天服务，还是留到未来服务；
- **日内路径尺度**：已决定今天服务的订单，应如何分配给车辆、趟次和访问顺序；
- **仓库执行尺度**：已生成的路线是否会在门位、拣货或暂存层面引发不可执行高峰。

如果将三者直接压缩进一个 monolithic 模型，则会带来两个问题：

1. **计算复杂度过高**：跨日 assignment、多趟 routing、仓库资源约束同时建模会导致变量和约束急剧膨胀；
2. **诊断能力不足**：若结果较差，将难以区分究竟是 admission 错误、routing 结构不足，还是 depot-resource coupling 过强所致。

因此，本文采用的分层框架兼顾了：
- 工业级可计算性；
- 研究上可解释性；
- 论文实验上可拆解性。

### 3.1.2 End-to-end workflow

在任意规划日 `t`，框架按如下流程运行：

```text
Input state at day t
  -> Rolling Controller
  -> Daily Routing Backend
  -> Depot-aware Repair
  -> Daily execution plan + KPI + next-day state
```

更细化的执行逻辑如下：

1. 从订单池中读取当日可见订单；
2. 计算订单跨日优先级，并划分 protected / flex / deferred；
3. 将当日 admitted 订单送入单日 routing backend；
4. 构建 trip1 / trip2 可行路线；
5. 计算 gate / picking / staging / trip2 风险；
6. 对高风险解进行 repair；
7. 输出 assigned / deferred / failed 订单及 KPI；
8. 将未完成订单与 commitments 更新到下一日状态。

## 3.2 Rolling Controller

### 3.2.1 Role of the controller

Rolling Controller 的作用不是直接生成路径，而是控制**今天哪些订单应该进入路径求解器**。这一步是整个框架的关键，因为实际系统的崩溃往往并非来自 routing solver 本身，而是来自于：

- 当天 admission 过量；
- late-window 订单过多集中到危险时段；
- protected orders 被高吞吐 flex 订单挤掉；
- depot 执行压力被提前放大。

因此，controller 的核心使命可以概括为：

> 在服务率、未来 backlog 风险、仓库压力与计划稳定性之间取得平衡。

### 3.2.2 Inputs

控制层在规划日 `t` 接收如下输入：

- 当前已释放未完成订单集 `V_t`；
- 每个订单的释放日、最早/最晚可服务日；
- 客户时间窗、服务时长、拣货工作量；
- depot 与车辆资源摘要；
- 前一日状态，包括 carryover、历史承诺与失败记录；
- depot risk proxy，例如 late bucket 压力、trip2 压力、历史 overload 桶分布。

### 3.2.3 Outputs

控制层输出一个**当日订单决策结果**：

- `protected_orders`：必须优先落地的订单；
- `admitted_flex_orders`：允许今天一起尝试的灵活订单；
- `deferred_orders`：本日不进入 routing backend 的订单；
- optional：
  - `priority_tiers`
  - `route_budget_hints`
  - `trip2_caution_flags`

### 3.2.4 Order scoring

设订单 `i` 在规划日 `t` 的跨日优先级评分为：

`S(i,t) = U(i,t) + A(i,t) + R(i,t) + C(i,t) - D(i,t)`

其中：

- `U(i,t)`：deadline urgency，越接近最晚服务日越高；
- `A(i,t)`：waiting age，释放后已等待越久越高；
- `R(i,t)`：service risk，时间窗越紧、插入难度越高越高；
- `C(i,t)`：commitment / stability bonus，若前一日已承诺则提高保留倾向；
- `D(i,t)`：depot burden adjustment，若其进入今天会显著增大 depot 风险，则可适当扣分。

控制层并不需要把这个打分写成复杂精确优化，而应保持**轻量、可解释、易调参**。

### 3.2.5 Reservation and clipping

本文建议在控制层采用：

> **score + reservation + clipping**

具体包括：

1. **Protected reservation**：先为临近截止或历史承诺订单预留结构空间；
2. **Flex admission**：在 protected 之后填入 normal flex；
3. **Risk clipping**：对 late-window、trip2-heavy、depot-burst 倾向强的订单实施 admission 剪裁；
4. **Deferred carryover**：将剩余订单保留到未来日期。

### 3.2.6 Plan stability

由于系统是 rolling horizon，每天都可能重算，如果不显式考虑 stability，则会出现同一订单在相邻两天中反复被提前/推迟的情况。为此，本文建议对下列行为施加 penalty：

- 已承诺订单的再次延期；
- 前日计划中的执行日被频繁改动；
- 已进入 protected pool 的订单被重新降级。

这类 penalty 的目标不是绝对禁止重排，而是避免**不必要的计划震荡**。

### 3.2.7 Pseudocode

```text
function controller(day_state, controller_config):
    visible_orders = collect_released_unfinished_orders(day_state)
    scored_orders = score_orders(visible_orders, day_state, controller_config)

    protected = reserve_protected_orders(scored_orders, day_state, controller_config)
    flex_candidates = collect_remaining_flex_orders(scored_orders, protected)
    admitted_flex = clip_flex_by_capacity_and_depot_risk(flex_candidates, day_state, controller_config)
    deferred = visible_orders - protected - admitted_flex

    return ControllerDecision(
        protected_orders=protected,
        admitted_flex_orders=admitted_flex,
        deferred_orders=deferred,
        metadata=...
    )
```

## 3.3 Daily Rich VRP Backend

### 3.3.1 Role of the routing backend

单日 routing backend 的任务是：

> 在给定的当日 admitted 订单集上，构建一组满足多趟、时间窗、容量、工时与回仓约束的日内路线。

与经典 VRPTW 相比，这里有三个额外困难：

1. 每台物理车辆最多两趟，且 trip2 必须满足时间接力；
2. 订单来自 controller 的 mixed pool，既有 protected 也有 flex；
3. routing backend 需要为后续 depot repair 留出可修复空间，而不是一味追求局部最短距离。

### 3.3.2 Recommended solving paradigm

最推荐的求解范式为：

> **Constructive insertion + selective local improvement**

而不是一开始就做大规模黑箱 ALNS。

原因是：
- 它更适合和 controller 的 protected/flex 结构接口对接；
- 更容易显式记录失败原因；
- 更利于 thesis 中进行模块化 ablation；
- 更容易与 depot-aware repair 构成前后两阶段闭环。

### 3.3.3 Constraints handled in this layer

routing backend 至少保证以下约束：

- 每个订单最多服务一次；
- 服务开始时刻满足客户时间窗；
- 配送服务时间计入路线时长；
- 车辆三维容量不超载；
- 单趟路线不超过最大时长与最大距离；
- 路线需回仓；
- 每台物理车最多执行两趟；
- trip2 发车时间必须晚于 trip1 完成并加上装卸时长。

### 3.3.4 Internal modules

#### (1) Candidate sequencing
先决定订单插入顺序。推荐按如下优先级：
- protected orders first；
- tighter time windows earlier；
- older / more urgent orders earlier；
- optional：偏远订单单独提前插入，避免后期无位置。

#### (2) Seed route construction
为路线生成初始种子：
- 可以从 protected 订单开始；
- 可优先为偏远、时间窗紧、体量大订单建立 seed；
- trip1 一般先构造较稳主骨架，trip2 作为补充扩展。

#### (3) Insertion engine
对剩余订单逐步插入当前路线。每个候选插入动作的增量成本可由下式评价：

`delta = distance_cost + time_slack_penalty + trip2_risk + protected_displacement_penalty + depot_proxy_penalty`

其中：
- `distance_cost`：额外里程或时间；
- `time_slack_penalty`：压缩时间窗 slack 的代价；
- `trip2_risk`：若该插入促使路线更可能变成危险 trip2，则加罚；
- `protected_displacement_penalty`：避免 flex 挤掉 protected；
- `depot_proxy_penalty`：若插入显著恶化 depot late bucket，可提前施压。

#### (4) Trip-aware feasibility check
每次插入后都需更新：
- 路线时间轴；
- 车辆日内占用；
- trip1 / trip2 relay feasibility；
- return-to-depot timing。

#### (5) Local improvement
在构造完成后，可进行轻量局部搜索：
- relocate
- swap
- 2-opt / Or-opt
- intra-route departure adjustment（仅限 routing 层内能安全做的部分）

### 3.3.5 Failure labeling
无法服务的订单必须被明确标记原因。例如：
- `no_feasible_route`
- `time_window_blocked`
- `capacity_blocked`
- `trip2_blocked`
- `protected_space_missing`
- `controller_clipped`

这不仅是工程需要，更是论文诊断的重要基础。

### 3.3.6 Pseudocode

```text
function daily_routing(controller_decision, instance, routing_config):
    routes = initialize_empty_vehicle_day_plan(instance)
    candidate_orders = order_candidates(controller_decision)

    routes = build_seed_routes(routes, candidate_orders.protected, instance, routing_config)
    routes = insert_remaining_protected(routes, candidate_orders.protected, instance, routing_config)
    routes = insert_flex_orders(routes, candidate_orders.flex, instance, routing_config)
    routes = local_improve(routes, instance, routing_config)

    unassigned = collect_unassigned_with_reasons(...)
    return RoutingSolution(routes=routes, unassigned=unassigned, metadata=...)
```

## 3.4 Depot-aware Repair Layer

### 3.4.1 Motivation

routing-feasible 不等于 warehouse-feasible。即便所有客户时间窗、车辆容量与 trip relay 都可行，也仍可能出现：

- 某 15 分钟 bucket 出车数超 gate 上限；
- picking 某时段超 `PickingCapacityColliPerHour`；
- staging peak 超 `MaxStagingSpace`；
- 大量 trip2 聚集在危险时段，造成仓配联动高峰。

因此，本文将仓库执行层可行性单独建模为第三层 repair，而不是完全挤进 routing 主过程。

### 3.4.2 Why post-optimization repair is appropriate

若直接在构造阶段强硬引入所有 depot 约束，常会带来：
- admission 过早被压死；
- 路由吞吐显著下降；
- runtime 增长；
- 很难区分是结构性 admission 问题还是局部 depot timing 问题。

repair 层的价值在于：
- 先保 routing throughput；
- 再有针对性地修复 depot overload；
- 保持实验与论文叙事的清晰模块边界。

### 3.4.3 Diagnostics

repair 层首先对 routing solution 进行 depot diagnostics，包括：
- gate bucket usage；
- picking workload timeline；
- staging occupancy profile；
- trip2 distribution；
- overload bucket count；
- worst-bucket severity。

### 3.4.4 Repair neighborhoods

推荐的修复动作包括：

#### (1) Departure smoothing
不改变客户顺序，仅在允许区间内平移路线出发时刻，减少 gate 冲突。

#### (2) Route reassignment
将路线或客户转移给其他车辆/趟次，降低危险峰值。

#### (3) Tail split
把高风险尾部客户拆出，形成独立小路线或回退 deferred pool。

#### (4) Selective relocation / exchange
针对 worst overload buckets，仅对少量客户做 targeted swap 或 relocate。

#### (5) Selective rollback
若个别 risky flex 是 depot 爆峰主因，可从今日回退。

#### (6) Controlled refill
若 repair 释放了新的安全容量，可少量回补低风险 deferred flex。

### 3.4.5 Objective of the repair layer

repair 的目标不是单纯降里程，而是：

1. 尽量降低 depot penalty；
2. 减少 overload bucket 数与严重度；
3. 尽量保持已分配订单数；
4. 尽量少破坏 routing 层已形成的主体结构；
5. 保持合理计算时间。

### 3.4.6 Pseudocode

```text
function depot_repair(routing_solution, depot_model, repair_config):
    diagnostics = evaluate_depot_profile(routing_solution, depot_model)

    while repair_budget_remaining() and diagnostics.has_overload():
        candidates = generate_repair_moves(routing_solution, diagnostics, repair_config)
        best_move = select_best_move(candidates, diagnostics, repair_config)
        if not best_move:
            break
        routing_solution = apply_move(routing_solution, best_move)
        diagnostics = evaluate_depot_profile(routing_solution, depot_model)

    return RepairedSolution(solution=routing_solution, depot_diagnostics=diagnostics)
```

## 3.5 State Transition and Rolling Execution

### 3.5.1 State update

在每个规划日结束后，系统需要构建下一日状态。设：
- `Assigned_t` 为当日成功服务订单；
- `Deferred_t` 为未进入 routing 或被 rollback 的订单；
- `Failed_t` 为因过期或不可行被判定失败的订单。

则下一日状态 `State_{t+1}` 至少包括：
- 仍未完成订单池；
- 每单最新承诺状态；
- 历史服务/延期/失败标签；
- depot KPI 历史；
- optional：上一日路径结构摘要，用于 stability penalty。

### 3.5.2 Output artifacts

对于每个日 `t`，框架输出：
- 当日路线与 trip1 / trip2 调度结果；
- assigned / deferred / failed 订单列表；
- routing KPI；
- depot KPI；
- next-day carryover state。

## 3.6 Implementation mapping

为了让算法能够直接落实代码实现，本文将框架映射到以下软件模块：

- `controller/`：订单打分、reservation、admission clipping
- `routing/`：seed construction、insertion engine、trip-aware feasibility
- `repair/`：gate/picking/staging 诊断与修复
- `evaluation/`：service、routing、depot 指标
- `core/`：公共数据结构与滚动状态定义
- `io/`：benchmark、matrix、config 读取
- `experiments/`：Herlev / East / West 实验入口

## 3.7 Chapter summary

本章提出了一个适用于多日滚动仓配一体化多趟配送问题的三层求解框架。该框架通过将跨日 admission、单日路径构造和仓库资源修复进行分层建模，在保持工业级可计算性的同时，实现了研究上可解释、实验上可拆解、工程上可落地的系统结构。这一框架将作为后续算法实现与实验设计的统一主线。
