# 最优算法架构：多日滚动控制器 + 单日富约束路径后端 + 仓库资源修复层

## 1. 结论先行

对于当前 thesis 所定义的问题——**多日滚动规划、订单逐步释放、可跨日服务窗口、多趟车辆、仓库门位/拣货/暂存约束、真实路网矩阵**——最合适的不是一个把所有变量一次性塞进同一个模型的 giant MIP，也不是纯 RL 或端到端神经组合优化。

当前最优的总体架构应为：

> **Rolling-Horizon Controller + Daily Rich VRP Backend + Depot-Resource Repair**

中文可写为：

> **多日滚动控制器 + 单日富约束路径求解后端 + 仓库资源修复层**

它的核心思想不是“用一个算法直接解完整问题”，而是把问题拆成三个彼此耦合但职责清晰的层次：

1. **跨日控制层**：决定今天该接多少、哪些必须接、哪些可以留到后面
2. **单日路径层**：在当天订单集上构造多趟、带时间窗和容量约束的实际路线
3. **仓库修复层**：在不完全推翻路线的前提下，压低门位、拣货、暂存和 trip2 的执行风险

这套架构最适合 thesis 的原因在于：
- 它符合真实业务的滚动决策机制
- 它能清楚分离“跨日接单问题”和“单日路线问题”
- 它允许把 warehouse-resource 作为论文亮点纳入，但不会一开始就把 routing throughput 压垮
- 它既能解释 Herlev 这类较稳实例，也能解释 West 这类 admission 主导型困难实例

---

## 2. 为什么不推荐其他主架构

### 2.1 不推荐 giant MIP / monolithic exact model
不建议把以下所有内容一次性放进一个整体模型：
- 跨日分配
- 单日路径
- 多趟时序
- 仓库门位
- picking 吞吐
- staging 占用
- 大规模真实路网

原因：
- 组合规模过大，难以在 thesis 可接受时间内稳定求解
- 很难对大实例（如 east / west）形成可复现实验链路
- 一旦效果差，很难判断问题出在 admission、routing 还是 depot coupling

### 2.2 不推荐纯 RL / 纯神经组合优化
原因：
- 约束体系复杂，训练样本与 reward 设计成本极高
- 工业型硬约束不适合完全交给黑箱网络处理
- thesis 当前更需要可解释、可控、可拆解的结构化算法

### 2.3 不建议把完整 3D packing 当主线
如果当前数据只有：
- `ColliCount`
- `Volume`
- `Weight`

而没有稳定、完整、可信的：
- 单件长宽高
- orientation
- stackability
- LIFO 物理规则

那么更合适的口径是：

> 将车辆约束建模为多维容量限制，而不是完整三维装箱问题。

否则会显著增加复杂度，但未必带来同等研究收益。

## 3. 总体框架概览

整个求解流程按日滚动运行。在每个规划日 `t`，系统执行以下流水线：

```text
Step 0  读取状态
  - 当前已释放未完成订单
  - 前一日 carryover / deferred 状态
  - 仓库与车辆资源
  - 真实路网矩阵

Step 1  跨日控制层（Rolling Controller）
  - 评估订单时效、风险、仓库压力、未来机会成本
  - 决定今日 admitted / protected / deferred 订单集

Step 2  单日路径层（Daily Routing Backend）
  - 在 admitted 订单集上构建日内多趟路线
  - 保证基本 routing feasibility

Step 3  仓库资源修复层（Depot-Resource Repair）
  - 评估 depot overload profile
  - 修复 gate / picking / staging / trip2 冲突

Step 4  输出执行计划与状态更新
  - 今日 assigned / deferred / failed
  - 路线时间表
  - depot KPI
  - 滚动到下一天
```

---

## 4. Layer 1：跨日控制层（Rolling Controller）

这一层是全框架最关键的上层。它回答的不是“怎么走路”，而是：

- 今天应该放多少订单进 routing backend？
- 哪些订单必须今天处理？
- 哪些订单要被保护，不能被 flex 订单挤掉？
- 哪些晚窗、偏远、高 depot 风险订单今天应当少放甚至不放？
- 如何在服务率与未来崩盘风险之间做平衡？

### 4.1 输入
跨日控制层输入应包括：
- 当前日期 `t`
- 已释放但未完成订单池 `V_t`
- 每个订单的：
  - `DateRequested`
  - `DeliveryDateFrom`
  - `DeliveryDateFrom2`
  - 日内时间窗宽度
  - depot
  - `ColliCount / Volume / Weight`
  - `DeliveryTaskTime`
  - `PickingTaskTime`
- 历史状态：
  - carryover
  - 上一轮承诺计划
  - 累积 backlog
- 资源代理信息：
  - 当日车辆能力 proxy
  - 当日仓库能力 proxy
  - depot bucket 风险 proxy

### 4.2 输出
控制层输出不直接给出路线，而是给出**当日订单决策状态**：
- `must_serve_today`
- `protected_today`
- `admitted_flex_today`
- `defer_to_future`
- optional：
  - `priority tiers`
  - `route budget hints`
  - `trip2 caution flags`

### 4.3 推荐的控制逻辑
不建议在这一层使用重型精确优化。更推荐：

> **score + reservation + clipping**

即：
1. 先给每个订单打跨日风险分数
2. 再做 protected reservation
3. 最后按 depot/day 风险做 flex clipping

### 4.4 订单打分建议
可定义订单优先级分数：

`priority_score(i, t) = urgency + age + service_risk + isolation_penalty + depot_impact_adjustment`

其中各项可解释为：
- `urgency`：距离最晚服务日越近，分数越高
- `age`：释放时间越早、已等待越久，分数越高
- `service_risk`：时间窗越窄、插入越难，分数越高
- `isolation_penalty`：偏远且单独服务成本高的订单，可按策略上调或下调
- `depot_impact_adjustment`：若某类订单更可能造成 late bucket / trip2 / picking overload，则降低其当日 admission 倾向

### 4.5 控制层内部建议的四类订单分组
建议将订单显式分为：

#### A. Hard / Protected Orders
典型特征：
- 接近 deadline
- 多次延期后不能再拖
- 时间窗极紧
- 已在历史计划中承诺

这些订单优先进入 admitted 集。

#### B. Risky Flex Orders
典型特征：
- 可延期
- 但若放入今天，可能挤占 depot 关键资源
- 容易形成 late-window / trip2-heavy 压力

这些订单应该被控制性放入。

#### C. Normal Flex Orders
典型特征：
- 有一定弹性
- 路由兼容性较好
- 对总体 throughput 有正向贡献

这些订单应作为当天吞吐填充主力。

#### D. Deferrable Orders
典型特征：
- 时间窗尚宽
- 今天的边际收益不高
- 当前仓库/车队压力又偏大

这些订单可保留到未来。

### 4.6 capacity reservation 机制
控制层不应把全部当日容量让给 flex 订单。建议显式保留：
- `protected capacity reserve`
- `late-window reserve`
- optional：`same-day recovery reserve`

这可以避免高吞吐 flex 订单把 hard orders 的结构空间完全吃掉。

### 4.7 plan stability 机制
滚动规划不能每天剧烈改计划。建议控制层目标中纳入：
- re-commit penalty
- reassignment penalty
- promised-order shift penalty

这样能维持执行层稳定性，也更符合真实运营。

### 4.8 控制层的目标不是最短路
它的目标应表述为：
- 提高窗口内按时服务率
- 降低逾期与失败风险
- 控制 carryover 雪球
- 避免 depot 资源崩盘
- 保持滚动计划稳定

## 5. Layer 2：单日富约束路径后端（Daily Rich VRP Backend）

这一层接收控制层产出的当日 admitted 订单，解决：

> 在今天这一个执行日里，如何把这些订单分配到车辆与 trip 上，并确定可执行的访问顺序与时间表。

### 5.1 建议定位
论文里建议把它写成：

> **A heuristic daily rich VRP backend for multi-trip routing with time windows and vehicle constraints**

不一定非要押注“纯 ALNS”。更稳妥的说法是：
- 主体为构造式启发式
- 可辅以局部搜索 / ALNS / VNS 等改进过程

### 5.2 这一层必须保证的约束
- 每单最多服务一次
- 客户时间窗可行
- `DeliveryTaskTime` 计入路线时间
- 三维容量可行：
  - `ColliCount`
  - `Volume`
  - `Weight`
- 单趟路线满足：
  - `MaxRouteDuration`
  - `MaxRouteDistance`
- 路线需回仓
- 每台物理车最多两趟
- `trip2` 必须在 `trip1` 完成后，且加上卸货与再装货时间

### 5.3 推荐内部结构
最推荐的不是一上来做庞大 ALNS，而是：

> **Constructive insertion + selective local repair**

即：
1. 初始 seed route 构建
2. protected-first 或 service-first 插入
3. throughput-oriented flex 填充
4. trip2-aware feasibility check
5. 小规模局部优化
6. 输出 routing-feasible 结果到 depot repair

### 5.4 单日后端的详细模块划分

#### 5.4.1 Candidate generation
先生成候选订单序列：
- protected 优先
- 同 depot 聚类
- 时间窗兼容性优先
- 可选加入 proximity / insertion gain 评价

#### 5.4.2 Seed route construction
为每台车的 trip1 / trip2 创建基础骨架。可采用：
- protected 订单优先成种子
- 紧时间窗订单优先成种子
- 偏远订单作为独立 seed，避免后续无处安放

#### 5.4.3 Insertion engine
对未插入订单执行逐个或批量插入。插入评价应综合：
- 额外行驶时间 / 距离
- 时间窗 slack 消耗
- trip2 时序风险
- depot bucket 风险 proxy
- protected displacement 风险

建议采用多种插入策略并行测试：
- greedy insertion
- regret-2 / regret-3 insertion
- service-first insertion
- throughput-first insertion

#### 5.4.4 Trip-aware routing logic
必须显式建模 trip1 / trip2：
- 每台车维护日内时间轴
- trip2 只能在 trip1 return + unload + reload 之后启动
- 若 trip2 导致晚窗集中或 depot 高峰恶化，应提高惩罚或直接禁止

#### 5.4.5 Failure labeling
无法插入的订单不能消失，必须分类：
- `no_feasible_route`
- `time_window_blocked`
- `capacity_blocked`
- `trip2_blocked`
- `depot_risk_clipped`
- `protected_space_missing`

这对 thesis 诊断非常关键。

### 5.5 单日后端的目标
这层主要追求：
- routing feasibility
- 高服务吞吐
- 尽量保住 protected orders
- 不过早制造严重 depot overload

也就是说，这层不要求一次性把所有仓库资源问题解决到最优，而是要输出一个：

> **高服务、基本路由可行、可被 depot repair 进一步修正的中间解。**

## 6. Layer 3：仓库资源修复层（Depot-Resource Repair）

这是当前 thesis 与普通 VRPTW 最大的区别之一。

如果只做 routing feasibility，不看 warehouse-resource，那么结果可能表面上服务很多单，但执行层会直接崩掉：
- gate 冲突
- picking 峰值爆仓
- staging 占用过大
- trip2 集中挤到危险时段

因此需要独立的第三层。

### 6.1 为什么 repair 层要独立存在
如果一开始就把所有 warehouse-resource 约束重压进主构造器，常见后果是：
- service throughput 显著下降
- 构造器过于保守
- runtime 变长
- 很难分辨到底是 routing 难还是 depot risk 难

把它独立成 repair 层有三个优点：
- 先得到高服务的 routing-feasible 解
- 再有针对性地下压 depot overload
- 模块边界清晰，更适合写 thesis 实验

### 6.2 输入
repair 层输入：
- 当日路线方案
- 每条路线的发车时间、回仓时间、trip 编号
- picking workload 映射
- gate bucket usage
- staging proxy / exact profile
- overload diagnostics

### 6.3 输出
repair 层输出：
- 调整后的路线与发车时刻
- route reassignment / split 结果
- 订单 rollback / refill 结果
- depot overload KPI

### 6.4 重点修复对象
建议修复层重点关注：

#### A. Gate overload
同一 15 分钟 bucket 出车数超过 `Gates`

#### B. Picking overload
同一时间段内 `PickingCapacityColliPerHour` 或 `PickingCapacityVolumePerHour` 超标

#### C. Staging overload
预拣后待装车货量超过 `MaxStagingSpace`

#### D. Trip2 clustering risk
大量 second-trip 路线集中到危险晚段，造成仓库与执行链条同时拥挤

### 6.5 repair 层推荐动作集合

#### 6.5.1 Departure smoothing
在不改客户顺序的前提下平移整条路线出发时间，降低 gate bucket 冲突。

#### 6.5.2 Route reassignment
在车辆之间重新分配路线或客户，以平衡 trip2 压力和 depot 峰值。

#### 6.5.3 Tail split / route split
把晚段压力最大的尾部客户拆出，形成新路线或回退到 deferred 池。

#### 6.5.4 Selective relocation / exchange
围绕 worst overload buckets 进行小范围客户搬移或交换。

#### 6.5.5 Selective rollback
当某些 risky flex 订单是 depot 爆峰主因时，可将它们回退到 deferred 状态。

#### 6.5.6 Controlled refill
若某些修复后释放出安全容量，可从 deferred 池回补少量低风险 flex 订单。

### 6.6 repair 层目标
repair 的目标不应是单纯最短路，而应是多目标折中：
- depot penalty 最小化
- overload bucket 数最小化
- assigned orders 尽量保持
- routing feasibility 不崩
- runtime 可控

## 7. 三层之间的接口定义建议

为了让整个 thesis 后续代码与实验更清晰，建议把三层接口定义明确。

### 7.1 Controller -> Routing
输入给 Routing 的结构可以是：
- `protected_orders`
- `admitted_flex_orders`
- `deferred_orders`
- `routing_hints`

### 7.2 Routing -> Repair
输入给 Repair 的结构可以是：
- route list
- per-route departure / arrival timeline
- trip1 / trip2 mapping
- unassigned list with reasons
- preliminary depot usage profile

### 7.3 Repair -> Next-day state
输出给下一日滚动状态的结构可以是：
- assigned today
- deferred to future
- failed expired
- commitment record
- depot KPI history

这样写进 thesis，读者会非常清楚系统如何滚动运行。

## 8. 目标函数建议：分层而不是单一目标

### 8.1 Controller 层目标
建议优先级：
1. 保护临近截止订单
2. 抑制未来 backlog 崩盘
3. 控制 depot/day admission 风险
4. 减少计划震荡

### 8.2 Routing 层目标
建议优先级：
1. 最大化 routing-feasible assigned orders
2. 保住 protected orders
3. 控制距离/时间成本
4. 降低明显的 trip2 风险

### 8.3 Repair 层目标
建议优先级：
1. 降低 depot overload
2. 降低 worst bucket severity
3. 尽量保持 assigned orders
4. 尽量少改动已形成的路径结构

---

## 9. 论文中最适合强调的创新点

### 9.1 滚动视角下的跨日-日内分层架构
你的问题不是单日 VRPTW，而是：
- 订单逐步释放
- 窗口跨多日
- 今日决策会影响未来压力

所以明确引入上层 controller 是必要创新，而不是实现细节。

### 9.2 warehouse-resource 不是后置报表，而是闭环进入求解
这是仓配一体化的关键贡献点。

### 9.3 多趟闭环逻辑作为核心结构约束
不是简单地“车可以多跑一次”，而是 trip1 与 trip2 在同一物理车时间轴上的强耦合。

### 9.4 通过 failure labeling 解释不可行性来源
这会让 thesis 不只是“跑出结果”，而是能解释：
- 为什么某类实例失败
- 为什么 admission 比 routing 更关键
- 为什么 depot repair 在不同实例上作用不同

## 10. 推荐的实验组织方式

为了让论文叙事更清晰，建议实验分三组。

### 10.1 组一：基础 routing 对照
- 无 controller
- routing only
- 看吞吐、路线可行性、depot penalty

### 10.2 组二：controller 变体对照
- weak controller
- deadline protection
- protected reservation
- service-first admission
- high-admission vs conservative-admission

目标：解释 admission 层对服务率和后续 depot 风险的影响。

### 10.3 组三：depot repair 变体对照
- no repair
- smoothing only
- overload-focused reassignment
- split / rollback / refill variants

目标：解释 warehouse-resource 修复的真实边际作用。

## 11. 针对当前 thesis 的落地建议

结合当前问题定义与已有实验经验，最稳妥的主线不是“追求最复杂算法名词”，而是：

> **Controller-led rolling planning with a daily constructive routing backend and depot-aware repair**

中文可写为：

> **以跨日控制器为主导、以单日构造式多趟路径后端为执行核心、并由仓库资源修复层闭环校正的滚动规划框架。**

如果需要再简化成一句论文摘要里的表达，可以写为：

> 本文提出一种面向多日滚动仓配一体化配送问题的三层启发式求解框架，将跨日订单准入控制、单日多趟富约束路径生成与仓库资源修复分离建模，并通过滚动状态更新实现服务率、执行可行性与仓库负载之间的平衡。

## 12. 最终推荐架构图（文字版）

```text
Input Layer
  ├─ Orders: release dates, service windows, customer time windows
  ├─ Demand: colli, volume, weight, delivery task, picking task
  ├─ Depot resources: gates, picking capacity, staging, loading/unloading
  ├─ Fleet: vehicle count, capacities, route duration/distance, multi-trip
  └─ Road-network matrices + previous-day rolling state

Layer 1: Rolling Controller
  ├─ urgency scoring
  ├─ protected-order reservation
  ├─ flex admission clipping
  ├─ depot/day risk control
  └─ plan stability control

Layer 2: Daily Rich VRP Backend
  ├─ seed route construction
  ├─ protected-first / service-first insertion
  ├─ throughput flex fill
  ├─ trip1-trip2 feasibility checks
  └─ limited local improvements

Layer 3: Depot-Resource Repair
  ├─ gate bucket diagnostics
  ├─ picking/staging overload diagnostics
  ├─ departure smoothing
  ├─ route reassignment / split / rollback / refill
  └─ depot-feasibility improvement

Output Layer
  ├─ assigned / deferred / failed orders
  ├─ daily route schedules
  ├─ depot overload metrics
  └─ updated rolling-horizon state for next day
```

## 13. 一句话总括

**最优架构不是“纯 ALNS”或“一个大一统模型”，而是一个以 rolling controller 为上层决策核心、以单日多趟富约束 routing heuristic 为执行后端、并由 depot-aware repair 闭环修正仓库资源冲突的三层滚动求解框架。**
