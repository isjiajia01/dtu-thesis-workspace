04 — 22.02 Big Picture

The whole solver picture in 22.02

22.02 is not just a daily routing solver. It is a rolling-horizon multi-day method with one main heuristic line, one strengthening layer, and one exact-support layer.

Main active line two-level ALNS
Rolling horizon

Solve several days together, then commit only the current day.

Cross-day search

Move flexible orders across feasible service days.

Within-day rebuilding

Rebuild daily routes and evaluate the updated multi-day incumbent.

Support layers RSP + exact track
RSP intensification

Recombine good current-day routes found during ALNS.

Restricted master

Improve the incumbent if a better route-pool combination exists.

Exact support

Column generation / pricing is used for bounds and appendix-level support, not as the main operational solver.

One-sentence summary: in 22.02, ALNS is the main search engine, RSP is an optional current-day polishing step, and the exact line is mainly for lower bounds and technical support.

05 — Current-Day Route Pool

A simple example closer to the current 22.02 code

ALNS searches a multi-day solution, but the current implementation mainly sends today's routes into the RSP pool.

Setup: today is Monday. Two vans are available today. Orders o1, o2, o3 can still be served today, while o4 can be pushed to Tuesday if needed. The useful case is when ALNS stores several good Monday routes, but its current incumbent still keeps one order for Tuesday.

ALNS Iteration A current day = Mon
  • Mon: route rA = [o1, o2], cost 38
  • Tue: later route set [o3, o4], cost 54
  • Pool contribution: save only rA for current-day intensification
ALNS Iteration B different day assignment
  • Mon: route rB = [o3], cost 10
  • Tue: later route set [o2, o4], cost 52
  • Pool contribution: add rB to the Monday route pool
ALNS Iteration C another variant
  • Mon: route rC = [o1, o3], cost 31
  • Tue: later route set [o2, o4], cost 53
  • Pool contribution: add rC to the Monday route pool

What enters RSP here? not the whole Monday-Tuesday-Wednesday plan. The pool is closer to {rA(38), rB(10), rC(31)}: candidate current-day routes collected while the broader multi-day ALNS search runs.

Read this carefully: RSP does not receive Tuesday routes here. It only sees a small menu of Monday columns and tries to choose the best compatible subset.

06 — Incumbent Evolution

Walk through one multi-day incumbent story

The point is not that RSP sees the whole horizon as one master. The point is that ALNS keeps changing the multi-day incumbent, while today's routes accumulate in a reusable pool.

Incumbent after Iteration A cost = 122
  • Mon: rA = [o1, o2] cost 38
  • Tue: [o3, o4] cost 54
  • Penalty/other terms: 30

First feasible multi-day incumbent.

Incumbent after Iteration B cost = 111
  • Mon: rB = [o3] cost 10
  • Tue: [o2, o4] cost 52
  • Penalty/other terms: 28

Better cross-day arrangement, better incumbent.

Incumbent after Iteration C cost = 109
  • Mon: rC = [o1, o3] cost 31
  • Tue: [o2, o4] cost 53
  • Penalty/other terms: 27

Best ALNS incumbent so far is now 111, with Monday still using only rC.

Important: ALNS keeps the full multi-day incumbent, but the reusable route pool for RSP is only the Monday side: {rA, rB, rC}.

So the division of labor is: ALNS changes the full horizon incumbent; RSP only tries to improve the current-day slice of that incumbent.

07 — RSP Selection Example

Then RSP solves a restricted current-day master

Assume Monday has two route slots. RSP only chooses from the routes that already exist in the current-day pool.

Pool given to RSP Monday only
Column 1

rA = [o1, o2], cost 38

Column 2

rB = [o3], cost 10

Column 3

rC = [o1, o3], cost 31

RSP decision restricted master
Feasibility rule

Do not cover the same order twice, and do not exceed the two Monday slots.

Selected Monday plan

choose rA + rB. Together they cover o1, o2, o3 on Monday with total route cost 48.

Updated objective

ALNS incumbent had Monday rC only and kept o2 for Tuesday. RSP can improve the full incumbent from 111 to 104 by replacing Monday with rA + rB.

RSP model on this slide: choose binary variables x_A, x_B, x_C \in {0,1} to minimize 38x_A + 10x_B + 31x_C, subject to: each Monday order is covered at most once, and at most two Monday route slots are used.

Why does RSP pick rA + rB? rA covers o1,o2, rB covers o3, they do not overlap, and together they serve all Monday demand in two slots. By contrast, rC alone covers only o1,o3, so o2 is still pushed to Tuesday.

Reading this correctly: RSP does not invent new Monday routes and it does not optimize Tuesday or Wednesday here. It only recombines the existing current-day columns and then plugs that improved Monday plan back into the broader incumbent.

08 — From 111 To 104

How the full incumbent improves after RSP

RSP only changes the Monday slice, but that also changes what still needs to be served on Tuesday.

Before RSP ALNS incumbent = 111
Monday

rC = [o1, o3], cost 31

Tuesday

route [o2, o4], cost 53

Other terms

27, so total = 111

After RSP updated incumbent = 104
Monday

choose rA + rB = [o1, o2] + [o3], cost 48

Tuesday

only [o4] remains, so toy route cost drops to 29

Other terms

27, so total = 104

Arithmetic: before RSP, 31 + 53 + 27 = 111. After RSP, 48 + 29 + 27 = 104.

Why does Tuesday drop? because o2 is no longer deferred to Tuesday. Once Monday absorbs o2, Tuesday only needs to serve o4.

09 — What RSP Actually Does

RSP intensifies today's plan, not the full horizon

The current implementation is narrower than the fully general story of mixing the best routes from all days.

What RSP does now implementation-faithful
Input

A pool of candidate routes for the current day.

Optimization

Solve a restricted master to choose a better compatible subset of those current-day routes.

Output

An improved today plan, which is inserted back into the broader multi-day incumbent.

What it does not yet do broader idea
Not a full horizon master

It does not jointly recombine Monday, Tuesday, and Wednesday routes in one full multi-day master problem.

Not global optimality

It is exact only over the restricted route pool, not over all possible routes.

Not the main loop

ALNS remains the primary search engine; RSP is a companion intensification step.

Precise thesis wording: in the current 22.02 code, RSP is best described as a current-day route-pool intensification module embedded inside a broader multi-day ALNS search.
