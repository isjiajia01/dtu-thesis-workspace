# 可信度审计 v1：Herlev / East / West 当前 rolling baseline 结果为什么偏高

## 1. 结论先行

当前 Julia strongest chain 已经形成了真实可运行的 multi-day rolling baseline，并且统计口径中的**unique served / carryover accounting bug 已修正**。因此，当前输出不再存在“同一订单被重复计入 served”的明显汇总错误。

但是，Herlev / East / West 当前得到的超高 service rate：
- Herlev：`1043 / 1044 ≈ 99.9%`
- East：`3268 / 3268 = 100%`
- West：`4546 / 4551 ≈ 99.89%`

**仍然应该被视为 optimistic model-baseline，而不是 thesis 最终可信 SR。**

原因不是统计重复，而是：
1. 当前 depot constraints 主要仍是 penalty / proxy 驱动，而不是 hard execution feasibility；
2. 当前 routing / repair 仍有若干偏松的业务口径；
3. 当前大实例结果说明“主架构非常强”，但还没有经过足够严厉的 realism audit 来证明这些 SR 是最终可接受的论文结论。

---

## 2. 已确认不是主要问题的项

### 2.1 Multi-day unique-order accounting 已修正
当前 multi-day runner 已做：
- `completed_ids`：已服务订单会从未来 visibility 中移除；
- `expired_ids`：过期订单单独统计；
- `service_rate`：基于 `unique_assigned_orders / total_orders` 计算；
- `cumulative_assigned_orders` 与最终 `unique_assigned_orders` 一致。

因此，当前结果**不是**由简单的重复计数 bug 造成的。

### 2.2 结果也不是“每天都把所有可见订单瞬间吃掉”
从 day-by-day 输出看，三个实例都存在：
- 当天 deferred
- 次日 carryover
- 后续逐步消化

因此，当前结果不是 trivial zero-backlog 假象，而是一个真实 rolling process 下的高 service 输出。

---

## 3. 为什么当前结果仍然偏乐观：最可疑的约束松弛点

下面按怀疑程度从高到低列出。

## 3.1 仓库资源约束当前仍是 penalty / proxy，而不是 hard feasibility
这是**最重要**的可信度风险。

### 当前实现
- `gate`
- `picking`
- `staging`

都已经进入 diagnostics / repair / controller feedback，但目前主要形态是：
- bucket-based profile
- overload penalty
- shift / reassignment / rollback repair

### 可疑点
当前 solver 允许先生成 depot-overloaded solution，再通过 penalty 去“尽量修”。这意味着：
- depot overload 不会像 hard infeasibility 那样直接阻止方案成立；
- 只要 penalty 没大到迫使 controller / routing 大幅退让，系统仍可能保持极高 SR；
- 尤其 East / West 这类大实例，即使多天里 repeatedly overload，只要最终 carryover 清空，SR 仍可能接近 100%。

### 审计判断
**高度可疑，且最可能解释当前 SR 为何偏高。**

---

## 3.2 Depot closing 当前只约束 departure，不约束 return / end-of-day execution envelope
### 当前实现
在 routing feasibility 中：
- `departure_min <= warehouse.closing_min`
- 但 route return 只受 `max_route_duration` / `max_route_distance` 约束

### 可疑点
如果业务真实口径里 depot 的日内资源或车辆回仓后的卸货 / 再装货也受更严格的日界或作业界限影响，那么当前实现会偏乐观。

### 审计判断
**中高风险。** 这不是一定错误，但需要和 thesis 最终业务口径明确对齐。

---

## 3.3 Picking / staging profile 仍是 route-level aggregate proxy，不是细粒度 warehouse schedule
### 当前实现
- picking 负荷按 route 聚合后投到桶里
- staging 以 route total volume 近似

### 可疑点
真实仓内作业可能存在：
- 并发约束更细
- 拣货先后与装车先后耦合
- staging 占用持续时间与 route structure 更复杂

当前 proxy 可能低估仓内冲突，尤其在 East / West 这种高峰实例中。

### 审计判断
**高风险，但属于“模型抽象偏松”，不是 coding bug。**

---

## 3.4 当前 repair 仍以局部修复为主，未强制执行更严的 depot feasibility closure
### 当前实现
repair 现在有：
- shift
- reassignment
- rollback

但仍是启发式、局部、预算受限的修复动作。

### 可疑点
如果最终结果仍保留 depot penalty > 0，却仍被视为“成功服务”，则高 SR 的业务含义应当谨慎解释：
- 这更像“服务能力强，但执行风险仍存在”
- 不应直接等价于“完全执行可行的 99.9% SR”

### 审计判断
**高风险。** 当前应更强调 trade-off，而非把 SR 当最终唯一指标。

---

## 3.5 当前 single-day route feasibility 对 execution realism 仍有一定简化
例如：
- customer TW 已约束 `service_start`
- trip1/trip2 relay 已考虑 unloading/loading
- real matrix 已接入

但当前仍未 fully 覆盖所有可能的执行细节，例如：
- 更复杂的车辆日内排班现实性
- 车种与任务匹配更细逻辑
- 某些业务规则是否要求更严的 late-day execution guard

### 审计判断
**中等风险。** 不是当前最大的解释因素，但仍会整体抬高 SR。

---

## 4. 哪些现象最可疑

## 4.1 East / West 在 depot penalty 明显不低的情况下仍接近 100% SR
### 例子
- East 首日 depot penalty `10061.52`
- East 第三日 depot penalty `15364.2`
- West 首日 overload buckets `21`
- West 第三日 overload buckets `21`

但最终：
- East `100%`
- West `99.89%`

### 解释
这表明“服务成功”与“warehouse execution fully clean”之间当前仍有 gap。也就是：
> 当前模型更像在回答“能否通过 deferred + repair 最终完成几乎所有订单”，而不是“是否严格无风险地完成几乎所有订单”。

### 审计判断
**这是最值得在论文里谨慎解释的现象。**

---

## 4.2 West / East 的 carryover 能在 horizon 末尾被几乎完全清空
这本身不是错误，但对于这么高的 SR，需要额外解释：
- 是实例本身容量非常充裕？
- 还是 depot penalty 没被硬化为真正的阻断约束？

当前更可能是两者共同作用，但第二点的影响不能忽略。

---

## 4.3 Herlev 的 99.9% 可能部分来自实例本身确实较容易
Herlev 数据显示：
- `529` 个订单为 1-day window
- `515` 个订单为 5-day window
- 平均 volume 约 `0.99`
- 每日理论 trip slots `34`
- 每日理论总 volume cap `336`

因此，Herlev 不是一个不可能接近满服务的实例。

### 审计判断
Herlev 高 SR 并不一定异常，但它**仍然需要在更严格 depot feasibility 口径下再验证一次**。

---

## 5. 哪些地方最需要收紧后再重跑

## Priority 1：把 depot overload 从 soft penalty 往更硬的 admission / acceptance rule 推
建议：
- 若 depot penalty 高于阈值，则 controller 更强剪裁；
- 对 route acceptance / refill acceptance 引入更强 depot-safe 条件；
- 对最终结果引入“depot-feasible”与“service-high”双口径，而非只报 SR。

## Priority 2：对 picking / staging 引入更强的 hard-like guard
例如：
- 某些严重 overload bucket 直接视为 unacceptable；
- refill 不允许把严重 overload 再推高；
- repair 若无法压回阈值下，则强制 rollback 更多 flex。

## Priority 3：对 East / West 建一个 stricter realism config 再跑一轮
建议建立：
- 更保守 controller admission
- 更强 risky-flex clipping
- 更低 repair 接受阈值
- 更严格 depot penalty closure

然后再比较：
- SR 降多少
- depot penalty 降多少

这会更有论文意义。

## Priority 4：明确区分两类结果口径
论文中建议同时报告：
1. **Model service baseline**：当前 strongest chain 的高 SR
2. **Execution-stricter baseline**：更严约束口径下的结果

这样就不会把 optimistic baseline 误写成最终 business-feasible result。

---

## 6. 当前可以怎么写进 thesis

### 当前可以安全写的表述
- 当前结构化启发式主线在三个实例上都能形成高覆盖的 rolling baseline；
- 当前 strongest chain 在 accounting-correct 口径下，Herlev / East / West 均取得 very high model service rates；
- 但由于 warehouse-resource constraints 目前仍以 proxy / penalty 驱动为主，这些结果应视为 optimistic baselines，而非最终 business-feasible claims。

### 当前不建议直接写的表述
- “算法已在真实严格业务约束下实现 99.9% / 100% service rate”
- “East / West 已被完全解决”
- “当前 strongest chain 已可直接视为 final thesis result”

---

## 7. 审计结论

### 结论一
当前超高 SR **不是由简单统计 bug 造成的**，而是当前 strongest chain 在现有模型口径下确实具有非常强的清 backlog 能力。

### 结论二
当前最可疑、最应被继续收紧的地方，是：
1. depot constraints 仍为 proxy / penalty 主导
2. picking/staging 尚未 harden
3. 高 SR 与非零 depot penalty 并存，说明 execution realism 尚未完全闭合

### 结论三
因此，这些结果最适合作为：
> **optimistic rolling baselines / thesis main starting point**
而不适合作为：
> **最终严格业务 claim**

---

## 8. 下一步建议
1. 建一个 stricter realism config，对 Herlev / East / West 再跑一轮对照；
2. 报两类结果：model-high-SR vs stricter-execution-SR；
3. 继续沿当前 `controller + routing backend + depot-aware repair` 主线推进，而不是切换到 giant MIP。
